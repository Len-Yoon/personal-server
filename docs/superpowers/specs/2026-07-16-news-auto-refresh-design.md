# 뉴스 화면 자동 갱신 설계

## 문서 정보

| 항목 | 내용 |
|---|---|
| 문서명 | 뉴스 화면 자동 갱신 설계 |
| 작성일 | 2026-07-16 |
| 기준 자료 | `crawler-worker` 뉴스 라우터, 뉴스 캐시 서비스, 검색 템플릿 |
| 목적 | 사용자가 직접 새로고침하지 않아도 새 뉴스가 화면에 반영되도록 개선함 |
| 범위 | 뉴스 목록 화면의 자동 확인 및 목록 갱신 |
| 제외 범위 | 서버 수집 로직, 스케줄러, RSS 주기 정책 |

## 핵심 요약

뉴스 화면이 열려 있는 동안 브라우저가 60초마다 현재 카테고리의 API를 확인함. 서버의 기존 5분 캐시 정책은 유지하며, 응답의 기사 목록이 변경된 경우에만 화면의 뉴스 카드와 수집 상태를 갱신함.

## 상세 설계

### 1. 데이터 흐름

1. 사용자가 `/category?category=...`에 접속함.
2. 서버가 기존과 동일하게 캐시 또는 RSS 수집 결과를 HTML로 반환함.
3. 화면이 표시된 뒤 60초 간격으로 `/api/category?category=...`를 요청함.
4. 응답 기사 목록의 식별값이 기존 목록과 다르면 뉴스 카드 영역을 갱신함.
5. API 오류, 브라우저 비활성 탭, 응답 형식 오류가 발생하면 기존 화면을 유지함.

### 2. 갱신 정책

| 항목 | 기준 |
|---|---|
| 브라우저 확인 주기 | 60초 |
| 실제 RSS 수집 주기 | 기존 `NEWS_REFRESH_INTERVAL_SECONDS` 기본 300초 유지 |
| 갱신 조건 | 기사 URL 또는 기사 목록 순서가 변경됨 |
| 중복 갱신 방지 | 변경이 없으면 DOM 수정하지 않음 |
| 비활성 탭 | `document.hidden`이면 요청하지 않음 |
| 오류 처리 | 기존 목록을 유지하고 다음 주기에 재시도함 |

### 3. 사용자 경험

- 전체 페이지를 새로고침하지 않아 입력 상태와 스크롤 위치를 유지함.
- 목록 상단의 수집 상태 문구를 새 응답 기준으로 갱신함.
- `새로 수집` 링크는 기존처럼 즉시 강제 수집하는 기능으로 유지함.

## 검토 결과

- 기존 서버 캐시와 API를 재사용하므로 RSS 제공처 요청량은 증가하지 않음.
- 뉴스 수집 및 스케줄러 영역을 수정하지 않아 작업 범위 제한을 준수함.
- API 오류 시 화면을 비우지 않으므로 일시적인 네트워크 장애에 안전함.

## 확인 필요 사항

- 브라우저가 장시간 백그라운드 탭의 타이머를 지연시킬 수 있으므로 정확한 60초 실행은 보장하지 않음.
- 방문자가 없는 동안에는 자동 수집하지 않음. 상시 최신 캐시가 필요하면 별도 서버 작업이 필요함.

## 후속 조치

- 뉴스 라우트 테스트에 자동 갱신 설정과 API 데이터 속성을 검증하는 회귀 테스트를 추가함.
- 구현 후 관련 테스트와 전체 테스트를 실행함.
